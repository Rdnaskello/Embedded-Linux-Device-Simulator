# TASK-AUDIT-0.2 — As-Is Description Template (Audit-Grade, v1.0)

Тип таски: Audit / Methodology / Zero-Knowledge / High-Assurance

---

## 1) Аудиторський статус таски
Ця таска є обовʼязковою методологічною фазою, яка:
- уніфікує формат усіх подальших As-Is аудитів;
- гарантує порівнюваність результатів між модулями;
- усуває субʼєктивність описів;
- мінімізує ризик пропуску аспектів реалізації.

❗ Без завершеної TASK-AUDIT-0.2 заборонено виконувати будь-які TASK-AUDIT-1.x, 2.x, 3.x.

---

## 2) Початкові умови (Zero-Knowledge Assumption)
На момент виконання TASK-AUDIT-0.2 вважається, що:
- аудитори не знають архітектури проєкту;
- аудитори не знають, які модулі “головні” чи “допоміжні”;
- аудитори не знають, які частини критичні;
- єдиним джерелом істини є Audit Index, створений у TASK-AUDIT-0.1.

Будь-які припущення про “типові” розділи аудиту не допускаються без явного обґрунтування.

---

## 3) Головна мета (Primary Objective)
Створити єдиний, обовʼязковий шаблон As-Is опису, який:
- застосовується до будь-якого виявленого модуля/компонента;
- не залежить від технології, ролі чи складності;
- змушує аудитора фіксувати критичні аспекти поведінки, навіть якщо вони здаються “очевидними”;
- відокремлює факти від висновків.

Шаблон має відповідати на питання:
“Як зафіксувати реальний стан будь-якої частини проєкту так, щоб її поведінку можна було відтворити з опису?”

---

## 4) Scope (Область охоплення)
TASK-AUDIT-0.2 НЕ аналізує код і НЕ описує жоден модуль.

Вона:
- визначає ФОРМУ, а не ЗМІСТ;
- визначає СЕКЦІЇ, а не конкретні реалізації;
- задає обовʼязкові кути огляду, під якими має бути описаний будь-який модуль.

---

## 5) Жорсткі заборони ❌
У межах TASK-AUDIT-0.2 заборонено:
- створювати шаблон “під FakeCPU”, “під MemoryBus” тощо;
- використовувати терміни, повʼязані з конкретною архітектурою;
- додавати секції типу “Problems”, “Improvements”, “TODO”;
- включати будь-які оціночні формулювання;
- змішувати As-Is з To-Be.

---

## 6) Вимоги до As-Is Template (Audit-Grade)
Шаблон ОБОВʼЯЗКОВО має включати всі секції “As-Is Template (Canonical Form)”.
Відсутність хоча б однієї секції = шаблон недійсний.

---

## 7) As-Is Template (Canonical Form)

# <Component Name> — As-Is State (ELSIM v0.3.0)

## 1. Responsibility (Factual)
- Що компонент робить за фактом (спостережувані дії), без “навіщо/чому”.
- Формулювати дієсловами дії: “parses…”, “stores…”, “writes…”, “computes…”, “throws…”.
- Якщо відповідь загальна — додати 1–2 конкретні приклади (“коли X → відбувається Y”).

## 2. Public API / External Contract
- Публічні інтерфейси: функції/класи/методи, CLI-виклики, entry-points, конфіг-поля/формати, протокольні точки входу.
- Вхідні дані: типи/формати/обмеження/діапазони (як факт).
- Вихідні ефекти: повернені значення, зміни стану, I/O, логування, побічні ефекти, видимі зовні події (як факт).

## 3. Internal State Model
- Який стан компонент зберігає (фактичні елементи стану).
- Які змінні/поля/структури формують стан.
- Які з них зберігаються між викликами.
- Додатково (як факт, якщо виявлено): статичний/глобальний стан, кеші, лінива ініціалізація, стан у файлах/на диску.

## 4. Runtime Behavior
Описувати у фіксованому форматі “Trigger → Preconditions → Steps → Observable effects → Post-state”.
- Нормальні сценарії виконання (мінімум 1, якщо компонент виконується взагалі).
- Якщо порядок дій визначений — фіксувати його кроками.
- Якщо порядок не визначений/не гарантується — фіксувати це як факт.

## 5. Error and Edge Behavior
- Поведінка при: некоректних даних, відсутніх залежностях/ресурсах, граничних значеннях, неочікуваних викликах, неконсистентному стані.
- Для кожного випадку фіксувати механізм сигналу (як факт):
  - exception (тип/повідомлення, якщо видно),
  - return value / error code,
  - assert/terminate,
  - silent ignore,
  - undefined / unspecified / не визначено.
- Якщо поведінка не визначена або не знайдена у scope — це фіксується як факт.

## 6. Test Coverage (As-Is)
- Які тести явно стосуються цього компонента (імена тестів, файли).
- Що саме вони перевіряють (фактично, без оцінок).
- Які аспекти, для яких не виявлено тестів у scope (формулювати як факт: “не виявлено тестів для …”).

## 7. Documentation Alignment
- Які документи описують цей компонент (файли/розділи).
- Які аспекти задокументовані.
- Які аспекти НЕ задокументовані (факт).
- Які аспекти задокументовані, але суперечать реалізації (факт, з посиланням на джерела).

## 8. Real Usage / Observed Usage
- Де компонент реально використовується (приклади, CLI, демо, інтеграційні тести, конфіги).
- Які конкретні виклики/сценарії використання зустрічаються.
- Без висновків про достатність або коректність.

## 9. Dependencies & Assumptions (Factual)
- Які зовнішні залежності/ресурси компонент використовує (як факт):
  - інші компоненти/модулі,
  - файли/шляхи,
  - час/таймер/годинник,
  - змінні середовища/параметри запуску,
  - мережеві/процесні ресурси (якщо є),
  - бібліотеки/рантайм-умови (лише якщо це видно з коду/зборки).
- Які припущення/інваріанти видно з реалізації:
  - допустимі діапазони,
  - обов’язкові поля,
  - порядок викликів, який вимагається,
  - “має бути ініціалізовано до X”, якщо це явно видно.

---

## 8) Обовʼязкові правила заповнення шаблону
Аудитор ЗОБОВʼЯЗАНИЙ:
- заповнювати кожну секцію; якщо даних немає — писати “не виявлено у scope” або “не визначено реалізацією” (залежно від ситуації);
- чітко відрізняти:
  - “не реалізовано” (підтверджено кодом/контрактом),
  - “реалізовано, але не задокументовано” (підтверджено кодом + відсутнє в docs),
  - “не знайдено у scope” (не було підтверджено);
- не використовувати оціночні формулювання (“погано”, “недостатньо”, “краще”, “правильно/неправильно”);
- не робити висновків/припущень про задум, якщо це не випливає з артефактів;
- не узагальнювати без прикладу: якщо твердження загальне — додати мінімум один конкретний приклад;
- для кожного суттєвого факту давати посилання на джерело:
  - файл:рядок (код),
  - назва тесту/файл тесту,
  - документ/розділ з docs,
  - пункт з Audit Index.

---

## 9) Acceptance Criteria
TASK-AUDIT-0.2 вважається виконаною, якщо:
- створено єдиний канонічний As-Is шаблон (цей документ);
- шаблон не містить архітектурних припущень та оціночних формулювань;
- шаблон застосовний до будь-якого компонента;
- два різні аудитори, використовуючи цей шаблон, створять порівнювані As-Is описи одного компонента.

---

## 10) Роль у загальному аудиті
TASK-AUDIT-0.2:
- стандартизує мову аудиту;
- робить результати зіставними;
- дозволяє будувати повний As-Is образ проєкту;
- дозволяє пізніше механічно вивести список To-Be задач;
- є ключовою умовою для бойового планування без пропусків.

---

## Гарантія
Якщо TASK-AUDIT-0.1 і TASK-AUDIT-0.2 виконані строго:
- жоден компонент не буде пропущений;
- жодна поведінка не “загубиться між рядків”;
- формування задач на “бойову готовність” стане детермінованим процесом, а не інтуїтивним.
